# ML-модуль: сообщение -> память компании -> LLM -> решение

Модуль реализует локальный контур умного ответа без внешних API-ключей:

1. получает сообщение и историю диалога;
2. берёт `tenant_id` только из access JWT;
3. ищет релевантные готовые `KbChunk` этого tenant в PostgreSQL;
4. собирает prompt с профилем компании и `TenantAIConfig.system_prompt`;
5. вызывает настроенный LLM-провайдер;
6. считает confidence;
7. возвращает решение `auto_reply` или `escalate`.

## Текущий режим

По умолчанию используется `MockLLM`. Он не делает сетевых запросов и формирует
детерминированный ответ только из найденного контекста. YandexGPT и GigaChat
оставлены за общим интерфейсом, но реальные вызовы в этой задаче не реализуются
и не тестируются.

Неизвестное значение `LLM_PROVIDER` считается ошибкой конфигурации и не
подменяется mock-провайдером молча.

## HTTP-контракт

`POST /api/v1/ml/answer` и `POST /api/v1/knowledge/ask` принимают:

```json
{
  "message": "Сколько стоит подключение?",
  "history": [
    {"role": "customer", "text": "Здравствуйте"}
  ]
}
```

Клиент не может передать `tenant_id`, произвольную память, system prompt,
порог confidence или разрешение автоответа. Эти данные берутся из JWT, базы
знаний и `TenantAIConfig`.

Ответ содержит:

- `answer`;
- `confidence`;
- `decision`: `auto_reply` или `escalate`;
- `sources` из базы знаний tenant;
- `provider`;
- `used_context`.

`memory_override` существует только во внутреннем сервисном контракте и
используется в unit-тестах без базы данных.
